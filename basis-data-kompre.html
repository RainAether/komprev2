Model Data Konseptual (ERD)

Entitas, Atribut, Relasi — minim redundansi

💡 Kata kunci ujian: Sebutkan entitas, tentukan atribut + primary key, gambar relasi beserta kardinalitasnya (1:1, 1:N, M:N).
🧩 Komponen ERD yang Harus Hafal
  • Entitas — objek/benda nyata yang disimpan datanya. Contoh: Mahasiswa, Mata Kuliah, Dosen.
  • Atribut — properti dari entitas. Contoh: nim, nama, alamat. Atribut yang menjadi identifier disebut Primary Key.
  • Relasi — hubungan antar entitas. Ditulis dengan kata kerja. Contoh: Mahasiswa mengambil MataKuliah.
  • Kardinalitas — berapa banyak hubungan yang bisa terjadi (1:1, 1:N, M:N).

1️⃣ Relasi 1:1 (Satu ke Satu)

Satu record di tabel A hanya berhubungan dengan SATU record di tabel B.

Contoh: Pegawai ↔ KartuKaryawan
🔢 Relasi 1:N (Satu ke Banyak)

Satu record di tabel A berhubungan dengan BANYAK record di tabel B.

Contoh: Dosen → Mahasiswa (1 dosen punya banyak mahasiswa)
🔀 Relasi M:N (Banyak ke Banyak)

Banyak record di A berhubungan dengan banyak record di B. Perlu tabel perantara!

Contoh: Mahasiswa ↔ MataKuliah → tabel KRS
⚠️ Redundansi — Harus Dihindari!

Data yang sama disimpan berulang-ulang. Solusinya: pisahkan ke tabel berbeda dan hubungkan dengan relasi.

Simpan nama_dosen di tabel MK → redundan! Gunakan FK ke tabel Dosen.

Normalisasi Data (1NF → 2NF → 3NF)

Proses sistematis menghilangkan redundansi dan anomali

💡 Cara jawab di ujian: Tunjukkan tabel awal yang bermasalah → jelaskan masalahnya → ubah step by step sampai 3NF.
🔴 Tabel SEBELUM Normalisasi (Unnormalized)
nimnamakode_mknama_mknilainama_dosen
M001Budi BD101Basis DataADr. Andi
M001Budi SI201Sistem InformasiBDr. Siti
M002Rina BD101Basis DataADr. Andi
Masalah: nama dosen & nama MK disimpan berulang. Kalau nama dosen berubah, harus update banyak baris = update anomaly!
1️⃣ 1NF — First Normal Form
  • Semua nilai harus atomik (tidak boleh ada nilai berupa list/grup)
  • Setiap baris harus unik (ada primary key)
  • Tidak ada kolom berulang (misal: mk1, mk2, mk3)
Ciri pelanggaran: ada sel yang berisi "BD101, SI201" dalam 1 sel → pecah jadi baris terpisah!
2️⃣ 2NF — Second Normal Form
  • Sudah 1NF
  • Tidak ada partial dependency (ketergantungan sebagian)
  • Setiap atribut non-key harus bergantung pada seluruh primary key
Masalah muncul jika PK komposit. Contoh: PK = (nim, kode_mk), tapi nama_mk hanya bergantung ke kode_mk saja.
3️⃣ 3NF — Third Normal Form (TARGET UJIAN)
  • Sudah 2NF
  • Tidak ada transitive dependency (ketergantungan transitif)
  • Atribut non-key tidak boleh bergantung ke atribut non-key lain

3

DDL & DML — JOIN & Subquery

Implementasi perintah SQL secara tepat dan efisien

💡 Bedakan: DDL = mendefinisikan struktur (CREATE, ALTER, DROP). DML = memanipulasi data (SELECT, INSERT, UPDATE, DELETE).
🏗️ DDL — Data Definition Language
  • CREATE TABLE — membuat tabel baru
  • ALTER TABLE — ubah struktur tabel
  • DROP TABLE — hapus tabel permanen
  • CREATE INDEX — membuat index untuk mempercepat query
⚙️ DML — Data Manipulation Language
  • SELECT — mengambil/membaca data
  • INSERT — menambah data baru
  • UPDATE — mengubah data
  • DELETE — menghapus data

🏗️ Contoh DDL

DDL — CREATE TABLE
CREATE TABLE mahasiswa (
  nim     VARCHAR(10)  PRIMARY KEY,
  nama    VARCHAR(100) NOT NULL,
  alamat  TEXT,
  email   VARCHAR(100) UNIQUE
);

CREATE TABLE krs (
  nim      VARCHAR(10),
  kode_mk  VARCHAR(10),
  nilai    CHAR(1),
  PRIMARY KEY (nim, kode_mk),              -- PK komposit
  FOREIGN KEY (nim)     REFERENCES mahasiswa(nim),
  FOREIGN KEY (kode_mk) REFERENCES matakuliah(kode_mk)
);

🔗 JOIN — Menggabungkan Tabel

DML — INNER JOIN & LEFT JOIN
-- INNER JOIN: hanya tampilkan yang ada di KEDUA tabel
SELECT
  m.nim, m.nama,
  mk.nama_mk,
  k.nilai
FROM       mahasiswa  m
INNER JOIN krs        k   ON m.nim     = k.nim
INNER JOIN matakuliah mk  ON k.kode_mk = mk.kode_mk;

-- LEFT JOIN: tampilkan SEMUA mahasiswa,
-- meski belum ada KRS (nilai = NULL)
SELECT m.nama, k.kode_mk, k.nilai
FROM      mahasiswa m
LEFT JOIN krs       k  ON m.nim = k.nim;

🔍 Subquery — Query di Dalam Query

DML — SUBQUERY
-- Tampilkan mahasiswa yang punya nilai A
SELECT nama
FROM   mahasiswa
WHERE  nim IN (
  SELECT nim FROM krs WHERE nilai = 'A'
);

-- Mahasiswa yang IPK-nya di atas rata-rata
SELECT nama
FROM   mahasiswa
WHERE  ipk > (
  SELECT AVG(ipk) FROM mahasiswa
);
⚠️ Perbedaan JOIN vs Subquery: JOIN lebih efisien untuk menggabungkan tabel dan menampilkan kolom dari banyak tabel. Subquery cocok untuk kondisi filter yang bergantung pada hasil query lain.

Primary Key, Foreign Key & Referential Integrity

Konsep kunci perancangan basis data yang robust

💡 Ini sering keluar ujian! Hafal definisi, fungsi, dan contohnya. Jelaskan juga apa yang terjadi jika Referential Integrity dilanggar.
🔑 Primary Key (PK)
  • Kolom (atau gabungan kolom) yang mengidentifikasi setiap baris secara unik
  • Nilai PK tidak boleh NULL dan tidak boleh duplikat
  • Setiap tabel hanya boleh punya 1 Primary Key
  • Bisa komposit: PK terdiri dari 2+ kolom (contoh: nim + kode_mk)
nim VARCHAR(10) PRIMARY KEY
🔗 Foreign Key (FK)
  • Kolom yang merujuk ke Primary Key tabel lain
  • Digunakan untuk membuat relasi antar tabel
  • Nilainya harus ada di tabel yang dirujuk (atau NULL)
  • Tabel yang punya FK = child table; yang dirujuk = parent table
FOREIGN KEY (nim) REFERENCES mahasiswa(nim)
🛡️ Referential Integrity — Menjaga Konsistensi Data

Aturan yang memastikan nilai Foreign Key selalu valid — tidak boleh menunjuk ke baris yang tidak ada di parent table.

  • INSERT gagal — tidak bisa tambah data di child table jika nilai FK tidak ada di parent
  • DELETE gagal — tidak bisa hapus data di parent table jika masih ada child yang merujuk
  • UPDATE CASCADE — jika PK parent diubah, FK di child ikut berubah otomatis
  • DELETE CASCADE — jika parent dihapus, semua child terkait ikut terhapus
Contoh Implementasi FK + Referential Integrity
CREATE TABLE krs (
  nim     VARCHAR(10),
  kode_mk VARCHAR(10),
  nilai   CHAR(1),

  PRIMARY KEY (nim, kode_mk),

  FOREIGN KEY (nim)
    REFERENCES mahasiswa(nim)
    ON DELETE CASCADE     -- hapus mahasiswa → KRS ikut terhapus
    ON UPDATE CASCADE,    -- update nim → FK di KRS ikut update

  FOREIGN KEY (kode_mk)
    REFERENCES matakuliah(kode_mk)
    ON DELETE RESTRICT    -- TOLAK hapus MK jika masih ada di KRS
);
❌ Pelanggaran Referential Integrity:
INSERT INTO krs VALUES ('M999', 'BD101', 'A') → ERROR! Karena nim 'M999' tidak ada di tabel mahasiswa.
✅ Contoh Valid:
INSERT INTO krs VALUES ('M001', 'BD101', 'A') → SUKSES! Karena M001 ada di tabel mahasiswa dan BD101 ada di matakuliah.

🚀 Ringkasan Cepat Sebelum Ujian

📐 Indikator 1 — ERD

  • Entitas = objek, Atribut = propertinya
  • Primary Key = identifier unik tiap baris
  • 1:1 · 1:N · M:N (M:N butuh tabel perantara)
  • Hindari redundansi = pisahkan ke tabel beda

🔢 Indikator 2 — Normalisasi

  • 1NF = atomik, ada PK, tidak ada kolom ganda
  • 2NF = tidak ada partial dependency
  • 3NF = tidak ada transitive dependency
  • Hasil 3NF = tabel kecil-kecil, bersih

💻 Indikator 3 — DDL & DML

  • DDL = struktur: CREATE, ALTER, DROP
  • DML = data: SELECT, INSERT, UPDATE, DELETE
  • INNER JOIN = hanya yang cocok di kedua tabel
  • LEFT JOIN = semua kiri + yang cocok saja di kanan

🔑 Indikator 4 — PK, FK, Integrity

  • PK = unik, tidak NULL, identifikasi baris
  • FK = merujuk ke PK tabel lain
  • Referential Integrity = FK harus valid
  • CASCADE = ikut update/hapus otomatis
